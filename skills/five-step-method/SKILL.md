# 分级五步开发法（工具无关通用版）

服务对象：懂业务、懂产品，但不逐行读代码、代码能力完全依赖大模型的开发者。用户的控制点在 **Plan（业务审批）** 和 **Verify（证据验收）**，不在代码行。

设计取向：一线强模型（Claude Opus/Fable、GPT-5 级）代码能力已很强，只在模型天然薄弱处（顺从执行仓促指令、不主动验证、跳过流程）设最小闸门；规则宁删毋加，本文件每变长一分都是代价。这不是为弱模型时代设计的重流程，是给强模型配一副恰好够用的缰绳。

> 本文正文是**工具无关的算法内核**。凡是「切模型」「派子代理」「落盘路径」「触发方式」等与工具相关的动作，正文只写抽象意图，具体命令见文末**工具适配层**。采用前先照适配层把你的工具对应列读一遍。

## 第 0 步：任务分级（必做，先输出一句分级结论再动手）

| 级别 | 判据 | 流程深度 |
|------|------|---------|
| **A 微小** | 文案/样式/间距/单个明显小 Bug；不碰数据库、接口、权限 | 快速定位 → 最小修改 → 验证，不走完整五步 |
| **B 普通** | 新页面/表单/列表/普通接口/模块内功能迭代；不碰核心架构 | 五步全走，Review 用独立视角 |
| **C 高危** | 数据库结构、权限体系、支付/订单、公共模块重构、生产部署、多项目共用能力 | 五步全走 + 业务控制点检查 + 独立审查 + 人工批准 |

判级拿不准时 **就高不就低**。**同一会话中接连提出的每个后续需求与衍生子任务，逐个分级，不继承上一任务的级别**——连续任务最容易在这里静默脱轨。

## 第 0.5 步：模型档位前置确认（分级后、动手前）

强模型烧额度快、干流水活浪费；标准档足以应付执行型任务。分级后对照当前模型做一次匹配：

| 分级 | 当前是强档模型 | 动作 |
|------|--------------|------|
| A 级 | 是 | 不阻塞：直接干完，收尾附一行降档提醒 |
| B 级 | 是 | 停下不开工，提示用户降档或明确「继续」，等答复 |
| A / B 级 | 已是标准档 | 匹配，直接开工 |
| C 级 | 任意 | 直接进入流程，拍板点再按需提醒升档 |

这是纯防御闸门——不判断任务怎么做，只判断「用这个档位来做值不值」。具体切换命令见适配层。

## 第 0.8 步：需求碰撞（动手前的最后一道闸门）

大模型输出质量的上限由输入需求质量决定，而用户的第一句指令往往是仓促的直觉。**把仓促指令锤炼成完整需求是 AI 的责任，不是用户的**。

姿态四铁律：
1. **追问目标而非手段**——区分「用户说的做法」和「用户要解决的问题」，永远先确认后者。答案可能指向完全不同的方案
2. **有不同意见必须说**——判断用户方案非最优时必须直说 + 给备选 + 讲取舍；但用户坚持原方案则尊重执行，碰撞不是拒绝权
3. **能查就别问**——能从代码/文档/已有配置查到的，先查，不消耗用户时间
4. **每个问题必须自带推荐答案**——不许抛空问题让用户从零构思；先做判断给出推荐答案与理由，用户只需确认或推翻。想不出推荐答案 = 功课没做够，先补课再问

碰撞深度按分级：

| 分级 | 深度 | 节奏 |
|------|------|------|
| A 级 | 零碰撞 | 一句话复述理解，直接干 |
| B 级 | 轻量碰撞 | 打包问，一次最多 3 个，各附推荐答案 |
| C 级 | 完整碰撞 | 逐个问、一次一个，沿决策依赖下钻，形成《需求简报》落盘，用户确认后才继续 |

B 级打包问快、C 级逐个问深是刻意取舍：C 级决策之间常互相依赖，前一个答案会让后一个问题失去意义，只有一个一个问才能问准。

C 级完整碰撞必做四件事：反问采访（沿依赖逐支下钻、每问附推荐答案）／至少 1 个不同于原始指令的备选方案 + 取舍／补齐用户没提的边界（异常流、权限影响、数据兼容、回滚）／《需求简报》落盘（模板：业务目标 / 可验收成功标准 / 范围内外 / 已定关键决策 / 备选及否决理由 / 边界风险 / **尚未明确（迷雾区：当下答不了、待前置决策解锁的问题——不许硬问硬答造伪决策）**）。

Research 及之后各步以《需求简报》（C 级）或碰撞确认结论（B 级）为准，不再以用户原始指令为准。

## 超纲升级：任务大于单个会话时（先制图，再动工）

触发信号：碰撞或 Research 发现待决问题超 5 个且互相依赖、明显要分多期、牵涉 3 个以上子系统。此时**停止直接进五步，先制图**并告知用户「本任务超纲，建议先建地图」。

1. **地图文件**：五段结构——目的地 / 备注 / 已定决策（含理由与日期）/ **尚未明确（迷雾区）** / 范围外（落盘路径见适配层）
2. **决策票据**：每张票据 = 一个待决策项（「要不要、怎么做」，不是代码任务），标状态与依赖。迷雾区里还问不出精确问题的部分，等前置票据解决后再「毕业」成新票据
3. **Plan, don't do**：地图完成 = 迷雾区清空、无票待决——动手前没有任何事还需先决定
4. **一会话一票据**：每会话只认领一张票据（或一期实施），完成即更新地图、结束会话；下会话从地图继承上下文，不依赖会话记忆
5. 地图内每期实施照常按分级走五步；地图更新是该期收尾动作

## 五步执行

### 1. Research —— 先理解，不要直接写
- 先读项目文档（如 CLAUDE.md / AGENTS.md / docs/）和相关代码，找可复用能力和依赖
- 输出简短现状理解 + 不确定项，随 Plan 一并呈批；若发现推翻碰撞结论/简报前提的事实，立即停下报告，不得带着变更后的前提直接写 Plan
- B/C 级检索由广到深：广撒网 → 聚焦关键区 → 查遗漏和隐性依赖（不设轮数上限——模型的弱点是查得不够，不是查太多）

### 2. Plan —— 用户的主控制点
- 输出：目标 / 本次做什么 / 不做什么 / 改哪些文件 / 是否影响数据库·权限·API·公共组件·部署 / 如何验证 / 如何回滚
- **用业务语言写**，让不读代码的人能判断「流程对不对、范围有没有膨胀」
- C 级在此节点提醒升档审定（见适配层）
- **等用户明确批准才进入实施**

### 3. Implement —— 按批准的计划执行
- 只按批准计划做；最小改动；优先复用现有模块；不顺便重构无关区域；超出计划立即停下报告
- 每完成一项跑局部验证并记录，再继续下一项（多文件实施 / 纯抄写的分派方式见适配层）

### 4. Review —— 换视角检查，不许自评了事
- 用**独立视角**审查（不是刚写代码的那个上下文自说没问题）——B 级任务级审查、C 级独立终审（隔离方式见适配层）
- 检查清单：需求符合性 / 范围膨胀 / 无关文件改动 / 重复建设 / 权限与数据风险 / 错误处理 / 测试真实存在 / **鉴权粒度本身**（谁能调这个端点，不只验鉴权之后的判断）/ 同一模式在别处是否也有同款漏洞

### 5. Verify —— 拿证据证明完成
- 「已完成 / 应该能跑」不算验证；必须给证据，**每条附一句业务语言说明它证明了哪条业务目标**；能让用户亲自操作确认的（打开页面点一遍、看数据还在不在），**优先交用户做最终验收**：
  - 构建 / 类型检查 / 测试命令的实际输出
  - Web 改动真实打开页面操作
  - 数据真的落库、刷新后仍在
  - 不同权限账号无法越权
  - 旧功能未被破坏
- 没有证据不得宣称完成，明确列出「未验证事项」
- 新能力会让旧代码里从未被检验的隐藏假设现形——跨批次交互 bug 往往只有真实操作链路能测出，静态审查与单测测不出

| 抽象动作 | Claude Code | Codex |
|---------|-------------|-------|
| 显式触发本技能 | 说暗号「五步法」或技能自动匹配 | `$five-step-method`；暗号写进全局 `AGENTS.md` |
| 强制「先分级」 | 写进全局 `~/.claude/CLAUDE.md`（每轮常驻） | 写进全局 `AGENTS.md` |
| 切最强档 / 标准档 | `/model opus`（或 fable）/ `/model sonnet` | 切换 reasoning 档（如高/低推理） |
| 疑难升级先低成本咨询 | 配了 advisor 则先调 advisor（不计费到调用才产生）；无果再切模型 | 无对等机制则直接切高推理档 |
| 独立视角审查 | 派具名子代理（如标准档审查代理、最强档终审代理，模型钉死防漏写） | 开一个新会话，只贴 Plan + Diff、不贴实现过程，要求独立审查——用会话隔离代替代理隔离 |
| 多文件实施 / 纯抄写分派 | 派标准档 / 轻量档子代理 | 当前会话直接做（无原生子代理，不强行模拟） |
| 《需求简报》/ 地图落盘路径 | 项目内 `docs/`（无则 `.claude/`） | 项目内 `docs/decisions/`、`docs/maps/`（或 `.codex/`） |

模型名对应各家最新旗舰系列，随版本更新替换；「强档 / 标准档 / 轻量档」是相对概念，按你能用到的模型阶梯就位。

## 业务控制点（C 级实施前逐项自答，用户只对有风险的项拍板）

下列为**填空模板**——把示例项换成你自己项目的高危资产清单。AI 逐项自答并各附一句依据；用户只对答「是」或存在风险的项拍板「该后果能不能接受」。

- 是否影响现有核心业务流程 / 线上主要交易或转化路径？（示例：你的电商店铺、支付渠道）
- 是否影响数据库权限或数据结构？（示例：你的后端数据库、多租户隔离）
- 是否影响多组织 / 多项目 / 多角色的共用结构？
- 是否会新建与现有能力重复的接口 / 组件？
- 是否修改公共组件？影响面是否已列出？
- 是否触碰生产环境？回滚方案是什么？
- 哪些动作必须人工确认后才能执行？
