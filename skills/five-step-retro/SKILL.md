# 五步法复盘（自我迭代循环）

职责：给 five-step-dev 做新陈代谢——用真实运行数据发现流程问题，经批准后升级流程本身。禁止凭印象改规则。

## 前置

- 本复盘属判断密集型任务：若当前主模型不是 Fable / Opus，先提醒用户升档再开始
- 读取三份材料：
  1. `~/.claude/skills/five-step-dev/logbook.md`（运行日志）
  2. `~/.claude/skills/five-step-dev/SKILL.md`（当前算法文本）
  3. `~/.claude/skills/five-step-dev/archive/` 下最近一份归档末尾的复盘结论摘要（只读摘要段，不读原始日志；首轮无归档则跳过）
- 日志不足 5 条时提示样本太少、结论可信度有限，询问用户是否仍要继续

## 分析维度（固定六项，逐项过，不许跳）

1. **分级质量**：有没有漏报分级的任务？判级与实际风险是否相符——判 A 却返工多次 = 判轻了；判 C 却一路无碰撞点 = 判重了
2. **碰撞质量**：碰撞轮数与返工率的关系——碰撞多但用户嫌烦（跳过、不答、说"直接做"）= 过度；碰撞少且事后返工 = 不足
3. **切换提醒采纳率**：升档/降档建议被采纳还是被无视？总被无视的提醒说明设计不合理，该改提醒而不是怪用户
4. **Verify 证据完整度**：多少任务给了真实证据（命令输出、页面实操）vs 口头宣称完成？
5. **日志完整度**：有没有任务明显走了五步法却没留记录？（漏记本身就是流程执行力问题；缺席条目不自证，复盘时用 git 证据交叉核对：扫活跃项目近周期 git log 的 AI 协作提交（Co-Authored-By: Claude），与日志条目比对日期与主题，有提交无日志即为漏记缺口——顺带覆盖维度 1 的漏报分级）
6. **效率趋势**：返工次数与上一周期对比是升是降；并逐条核对上轮改动是否兑现其预期收益，未兑现的列为回滚候选（按开源仓库 commit 历史回退，回滚同样走逐条批准闸门）

## 产出：《复盘报告》

每条改进提案强制四要素，缺一不许提：

> **现象**（一句话）｜ **证据**（引用具体日志条目）｜ **改法**（SKILL.md 具体改哪段、怎么改）｜ **预期收益**（下周期哪个指标应改善）

无日志证据支撑的直觉性想法，单独列入「观察待验证」区，不作为本轮改动候选，留给下周期用数据检验。

## 闸门（不可绕过）

- 升级五步法本身 = C 级开发需求：提案逐条经用户批准，批准哪条改哪条
- 禁止在复盘会话中"顺手"修改任何未经批准的内容——升级闸门的动作不能绕过闸门
- 使技能文本更长更重、或削弱业务人控制点（Plan 审批 / Verify 验收）的提案默认否决；确要提出，必须在四要素外写明复杂度与 token 代价及取舍理由，由用户确认后方可采纳

## 落地（按批准结果执行）

1. 按批准的提案修改 five-step-dev/SKILL.md
2. 同步 365Skill 开源仓库：commit message 写明每条改动的理由（英文 Conventional Commits）
3. 日志归档：logbook.md 现有条目移入 `~/.claude/skills/five-step-dev/archive/YYYY-MM-DD.md`，主文件清空回表头
4. 归档文件末尾附一段本轮复盘结论摘要（改了什么、观察什么指标、本周期关键指标基线值如返工次数），供下轮对比

## 移植说明

落地第 2 步中的 365Skill 是作者的技能仓库，请替换为你自己的版本管理仓库；其余路径为 Claude Code 标准用户目录，跨机器通用。配套的 five-step-dev 技能见本仓库同级目录。
