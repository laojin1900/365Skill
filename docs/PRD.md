# 五步法体系 PRD：面向业务型开发者的 AI 协作流程

> 本文档描述 `five-step-dev` + `five-step-retro` 两个技能构成的完整系统。目标读者：不逐行读代码、但要指挥 AI 完成开发任务的产品/业务负责人，以及想把这套流程搬进自己环境的任何人。

---

## 一、产品概述

### 要解决的问题

当前一代大模型（Claude、GPT 系列等）的编码能力已经足够强，但强模型不等于好结果——**模型执行指令很快，但极少主动打磨指令本身**。用户给的第一直觉指令往往是仓促的，模型会忠实地把仓促指令执行到底，产出质量因此被死死摁在输入质量的天花板之下。

对于不逐行审代码的业务型使用者，这个问题被进一步放大：他们没有能力在代码层面发现问题，只能在事后通过"感觉不对"来纠错，纠错成本高、滞后严重。

### 目标用户

- 懂业务、懂需求，但不具备逐行审代码能力的产品/业务负责人
- 独自用 AI 完成大部分开发工作，缺少同事做同行评审
- 在多个项目间切换，需要一套可复用、不依赖临场发挥的协作纪律

### 核心主张

> 把"打磨需求""审查代码""验证结果"这三件事，从用户的隐性负担，变成 AI 的显性责任，并用流程闸门强制执行——而不是寄望于用户每次都记得要求。

---

## 二、问题陈述：三个真实痛点

这套体系不是理论设计，是从真实使用中反推出来的。三个核心痛点，均有本系统实际运行时发生过的案例支撑：

### 痛点 1：流程执行力不均衡——说过的规则会被自己遗忘

即便已经口头约定"重要任务要走完整流程"，AI 在处理具体请求时仍会因为"这活儿看起来不大"而跳过分级、直接动手。用户往往感觉不到某套流程"在运行"，因为它从未真正露面。

**根治方法**：把"分级结论必须先说一句"写成不可跳过的强制规则，且明确写入使用者的全局配置文件（而非仅仅存在于技能描述里）——技能描述是"AI 可能看见的菜单"，全局配置文件是"AI 每轮必须遵守的军令"，两者强度不同，必须用后者兜底。

### 痛点 2：模型档位与任务复杂度错配——用最贵的模型干最简单的事

顶级模型（如本系统作者使用的 Fable 5）能力强、消耗额度快；标准档模型（如 Sonnet）足以应付大部分执行型任务。如果没有前置检查，用户很容易在用顶级模型闲聊时顺手提一个简单需求，AI 照单全收，额度被无声浪费。

**根治方法**：分级结果与当前模型档位做匹配检查，档位过高时**停下来问，而不是直接开工**。

### 痛点 3：仓促指令封顶输出质量——AI 顺从执行，而非主动碰撞

用户的第一句话通常是解决方案而非问题本身（"帮我加个导出按钮"，而不是"我需要让客户看到自己的订单历史"）。AI 如果直接按字面执行，很可能建出一个技术上正确、但业务上文不对题的东西。

**根治方法**：执行前插入一道"需求碰撞"闸门，把"问清楚"的责任显式交给 AI，而不是依赖用户自己想清楚再问。

## 三、系统架构

整套系统由 **一条主链（five-step-dev）** 和 **一个反馈环（five-step-retro）** 组成：

```
                    ┌─────────────────────────────────────────┐
                    │              主链：five-step-dev           │
                    │                                           │
用户发起需求 ──▶ 第0步 分级 ──▶ 第0.5步 档位确认 ──▶ 第0.8步 需求碰撞 │
                    │                                           │
                    │   ──▶ Research ──▶ Plan（用户审批）──▶       │
                    │       Implement ──▶ Review ──▶ Verify     │
                    │                        │                  │
                    └────────────────────────┼──────────────────┘
                                              │ 每任务写一条运行记录
                                              ▼
                                     logbook.md（运行日志）
                                              │ 满 10 条 / 用户随时触发
                                              ▼
                    ┌─────────────────────────────────────────┐
                    │           反馈环：five-step-retro           │
                    │                                           │
                    │  读日志 ──▶ 六维度分析 ──▶ 证据支撑的提案     │
                    │       ──▶ 用户逐条批准 ──▶ 升级 SKILL.md     │
                    │       ──▶ commit 留痕 ──▶ 日志归档、清零     │
                    └─────────────────────────────────────────┘
```

**四道前置闸门 + 五步执行 + 一条反馈环**，缺一不可：闸门解决"该不该做、该怎么问"，五步解决"怎么做、怎么证明做完了"，反馈环解决"流程本身要不要变"。

## 四、机制详解

### 4.1 第 0 步：任务分级（必做）

一切开始于一句分级结论，不许跳过：

| 级别 | 判据 | 流程深度 |
|---|---|---|
| **A 微小** | 文案/样式/间距/单个明显小 Bug；不碰数据库、接口、权限 | 快速定位 → 最小修改 → 页面验证，不走完整五步 |
| **B 普通** | 新页面/表单/列表/普通接口/模块内功能迭代；不碰核心架构 | 五步全走，Review 派子代理 |
| **C 高危** | 数据库结构、权限体系、支付/订单、公共模块重构、生产部署、多项目共用能力 | 五步全走 + 业务控制点检查 + 独立审查 + 人工批准 |

判不准时**就高不就低**——错判为轻，代价是隐患漏检；错判为重，代价只是多问几句，两者不对等。

### 4.2 第 0.5 步：模型档位前置确认

分级结论出来后，立刻对照当前主模型做一次匹配检查：

| 分级 | 当前模型 | 动作 |
|---|---|---|
| A / B 级 | 顶级档（如 Fable/Opus） | **停下，不开工**，提示用户降档或明确表示继续 |
| A / B 级 | 标准档（如 Sonnet/Haiku） | 匹配，直接开工 |
| C 级 | 任何档位 | 直接进入流程，拍板点再按需提醒升档 |

这是一道纯防御性闸门——它不判断任务该怎么做，只判断"用这个档位的模型来做，是否值得"。

### 4.3 第 0.8 步：需求碰撞

这是整套体系里最反直觉、也最核心的一步：**把"问清楚"的责任从用户移交给 AI**。

四条姿态铁律：

1. **追问目标而非手段**——用户说的是"怎么做"，AI 要先确认"为什么"，因为答案经常指向完全不同的方案
2. **有不同意见必须说**——判断用户方案不是最优解时必须直说、给备选，不许揣着明白装糊涂地顺从；但**用户坚持原方案则尊重执行**，碰撞不是拒绝权
3. **能查就别问**——凡是能从代码、文档、已有配置里查到的，不该消耗用户的时间去回答
4. **每个问题必须自带推荐答案**——绝不抛出一个空白问题让用户从零构思；AI 必须先做出判断、给出推荐答案和理由，用户只需确认或推翻；**想不出推荐答案，说明功课没做够，先补课（回到第 3 条查证）再问**，不许带着空答案硬问

碰撞深度按分级走：

| 分级 | 深度 | 节奏 |
|---|---|---|
| A 级 | 零碰撞 | 一句话复述理解，直接做 |
| B 级 | 轻量碰撞 | 打包问，一次最多 3 个关键问题，各附推荐答案 |
| C 级 | 完整碰撞 | 逐个问、一次一个，沿决策依赖关系下钻，形成《需求简报》落盘，用户确认简报后才继续 |

B/C 两档节奏不同是刻意设计：B 级任务风险低，打包节省往返；C 级任务的决策之间往往互相依赖（前一个答案会让后一个问题失去意义），只有一个一个问才能问准。

### 4.4 五步执行

碰撞完成后，进入真正的执行主体：

| 步骤 | 核心动作 | 用户角色 |
|---|---|---|
| **1. Research** | 先读文档和代码，理解现状，列出不确定项；未确认前不进入 Plan | 旁观 |
| **2. Plan** | 用业务语言输出目标/范围/影响面/验证方式/回滚方案 | **主控制点之一**：必须明确批准才能继续 |
| **3. Implement** | 只按批准的计划做，最小改动，不顺便重构 | 旁观，超范围立即被叫停报告 |
| **4. Review** | 换一个视角检查（B 级派任务审查、C 级派独立终审），不许自己审自己 | 旁观 |
| **5. Verify** | 用真实证据（命令输出、页面实操、数据落库）证明完成，禁止口头宣称 | **主控制点之二**：靠证据而非信任来验收 |

这套体系刻意把用户的两个控制点放在 **Plan（业务审批）** 和 **Verify（证据验收）**，而不是代码行——因为这两处恰好是不懂代码的人也能做出高质量判断的地方。

### 4.5 模型切换提醒（三个信号）

| 信号 | 触发时机 | 建议动作 |
|---|---|---|
| 降档确认 | 第 0.5 步，档位过高 | 提示降档或明确继续 |
| 升档拍板 | C 级任务 Plan 审批点 | 建议切最强档审定方案，审完切回执行档 |
| 升档诊断 | 同一错误连续失败 3 次 | 建议切最强档诊断，切换前先把已试方案写成文件，防止长会话上下文压缩丢失背景 |

### 4.6 反馈环：five-step-retro（自我迭代循环）

这是整套系统与市面同类技能包最大的差异——**它会用真实运行数据改进自己**，而不是一份一成不变的静态文档。

**燃料**：Verify 收尾时强制写一条运行日志（不写不算完成），格式钉死 3~5 行：判级、碰撞轮数、切换建议采纳情况、结果、遇到的问题。

**节拍**：日志满 10 条，AI 主动提议复盘；用户也可随时用暗号手动触发，不设定时任务（复盘需要用户在场逐条拍板，无人值守的复盘只会产出半成品）。

**分析**：固定六个维度，逐项过、不许跳：分级质量、碰撞质量、切换提醒采纳率、Verify 证据完整度、日志完整度、效率趋势（返工次数、审批往返轮数的周期对比）。

**提案格式**：强制四要素——现象、日志证据、具体改法、预期收益。没有日志证据支撑的直觉性想法，只能进"观察待验证"区，留给下个周期用数据检验，不作为本轮改动候选。

**闸门**：升级流程本身视为 C 级开发需求，**逐条经人工批准**，不做任何自动放权——这是刻意设计：如果"升级闸门"的动作可以绕过闸门审批，整套系统的公信力就会塌陷。

**落地**：批准的改动写入 SKILL.md，commit message 说明每条改动的理由，日志归档进入下个周期，附一段本轮复盘摘要供下轮对比。

## 五、实施指南

### 前置条件

- 一个支持 `SKILL.md` 开放目录格式的 AI 编程工具（Claude Code 官方支持；其他工具见第七节适配说明）
- 能够创建全局级技能目录的权限

### 部署步骤

1. **安装两个技能文件**
   ```bash
   mkdir -p ~/.claude/skills/five-step-dev ~/.claude/skills/five-step-retro
   # 将本仓库 skills/five-step-dev/SKILL.md、skills/five-step-retro/SKILL.md
   # 复制或软链到上述目录
   ```

2. **初始化运行日志**：在 `~/.claude/skills/five-step-dev/` 下创建 `logbook.md`，写入表头格式说明（参考本仓库同名文件）。

3. **（可选，强烈建议）配置具名子代理**：为 Review/Implement 环节预建模型钉死的子代理，避免每次派单都要手写模型参数：
   - 一个执行型代理（标准档模型）
   - 一个任务审查代理（标准档模型）
   - 一个终审代理（最强档模型）
   - 一个机械抄写代理（轻量档模型）
   
   没有子代理机制的工具，可用"通用代理 + 显式指定模型参数"或"独立会话人工换视角审查"替代。

4. **写入全局强制规则**：在你的全局配置文件（Claude Code 为 `~/.claude/CLAUDE.md`，其他工具通常是 `AGENTS.md` 或等价文件）中加入：
   ```text
   任何开发需求（新功能、Bug 修复、页面/样式改动、数据库/权限变更、
   接口、部署、脚本、以及对配置与技能文件的修改）一律先按 five-step-dev
   技能报出一行分级结论（A/B/C）再动手。不报分级就动手 = 违规。
   ```
   这一步不可省略——技能描述只是"AI 可能看见的菜单"，全局配置文件才是"每轮必须遵守的强制指令"，只写前者会导致技能形同虚设（本系统作者曾亲身踩过这个坑）。

5. **（可选）设置触发暗号**：给"显式调用"设置一个好记的短语（例如中文"五步法"），写入技能描述的触发条件和全局配置文件的映射规则，降低记忆负担。

6. **替换业务控制点清单**：把 C 级业务控制点清单里的示例项（支付渠道、数据库、部署平台等）换成你自己项目的高危资产清单。

### 验证部署是否生效

新开一个会话，提一个小需求（例如"帮我改个文案"）。如果 AI 的第一句话是分级结论（例如"判级：A 级……"），说明规则生效；如果直接动手，说明全局配置文件没有正确加载，需要检查文件路径和格式。

## 六、真实案例

以下两个案例均取自本系统实际运行过程，可作为分级和碰撞深度的参照样本。

### 案例一：B 级——"帮这套技能写一份 PRD"

用户提出"做一个 PRD 和实施说明"的需求后：

1. AI 判级为 B（文档产出，不碰代码/数据库/权限，但涉及呈现形式和范围的真实判断，不是纯抄写）
2. 轻量碰撞，一次打包问 3 个问题，各附推荐答案：产出形式与受众、范围是否覆盖两个技能、是否整合跨工具迁移信息
3. 用户一次性确认全部推荐方案
4. Research（核对两份技能原文）→ Plan（本文档的大纲，含目标/范围/文件/验证/回滚）→ 等待批准 → Implement（撰写本文档）→ **Review（派子代理独立核对文档与技能原文的一致性，实际查出 2 处遗漏——姿态铁律的两条兜底条款未被完整复述），返工修正**

全程 1 轮碰撞、1 次返工（Review 阶段拦下）。这次返工恰好验证了"换视角检查"这一步的价值：撰写者本人很容易漏掉自己写漏的东西，独立审查才拦得住。

### 案例二：C 级——"设计五步法的自我迭代循环"

用户提出"能不能定期复盘、自我升级"的需求后：

1. AI 判级为 C（多项目共用的公共能力升级，属于流程算法本身的改动）
2. 完整碰撞，逐个问、一次一个，共 4 轮，依次锁定：数据采集成本是否接受、复盘触发节拍怎么定、改动权限是否需要人工逐条批准、整套机制放在哪里（独立技能 vs 并入本体）
3. 每一轮用户确认后才问下一轮，前序答案实际影响了后续问题的必要性（例如"人工全部批准"锁定后，就不再需要问"哪些改动可以自动应用"这类问题）
4. 产出《需求简报》文件落盘，进入 Research → Plan（含六步实施计划）→ 用户批准 → Implement → Verify（三条命令行核查 + 收尾写日志，第一条日志记录的正是"建立日志系统"这件事，形成自指闭环）

全程 4 轮碰撞、1 份需求简报、0 次返工。**但如实说明一处流程偏差**：C 级本应有的"独立审查"这一步，实际执行中被跳过了——直接从 Implement 走到了 Verify，没有派子代理做换视角检查。这处偏差本身没有引发问题，但按体系自己的标准，这是一次不完整的执行，恰好是留给 five-step-retro 下一轮复盘该抓的典型案例：**流程文档写得再完备，也不能替代真实运行时的执行力检查**。
